离线收益核算层我们不管，我们主要是将agent决策层里改成通用泛化的。现在里面定义了具体偏好能力字段，这些字段是从仿真数据API得到司机状态得到的字段吗？如果是的话可以理解，如果不是就属于硬编码的一种偏好类，但我喜欢是LLM可以依据语义提取出来的那种。

你的判断正确：这些具体偏好能力字段不是仿真 API 原生提供的，而是 Agent 内部人为定义的。因此它们属于一种偏好类硬编码。

API 不会直接返回：

这些字段是在 Agent 内部的 [StructuredConstraints](I:/AI竞赛/满帮物流/demo_docs_release_20260509/demo4/agent/constraints.py:130) 中自行定义，再由编译函数 [compile_atoms_to_constraints](I:/AI竞赛/满帮物流/demo_docs_release_20260509/demo4/agent/preference_v4.py:1037) 填充。

**应该保留什么**

不能让 LLM 自由发明任意字段，否则后续代码无法稳定计算。因此仍然需要一个有限结构，但这个结构应当是“可观测事实原子”，而不是“偏好类型”。

目前 [preference_dsl.py](I:/AI竞赛/满帮物流/demo_docs_release_20260509/demo4/agent/preference_dsl.py:31) 已经有较合适的事实白名单：

这些字段来自司机状态、货源查询结果或动作执行记录，属于通用能力，应当保留。

**例如**

“发货地或目的地是惠州的单不接”不需要增加 forbidden_endpoint_regions 字段。LLM 可以直接提取为：

后续统一使用 evaluate_predicate() 判断即可，无需新增“惠州限制”或“端点区域限制”代码。

**建议改造方向**

将偏好表达统一收敛为：

curfew、daily_rest_minutes、free_days_needed 等字段不再作为主逻辑入口。它们最多作为由原子规则自动生成的性能缓存，不能决定系统支持哪些偏好。

当前代码已经具备通用 DSL 基础，但仍保留了较多旧式专用字段和专用判罚分支。下一步应当把它们逐步迁移到统一谓词解释器和聚合计算器。
