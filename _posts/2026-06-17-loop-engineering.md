# 前言
又出了一个新词，已经从，prompt工程-context工程-memory工程-harness工程到了loop工程。
前面四个工程都可以视为对agent的激发、驾驭。到了loop工程，就将主角变换到人所承担的角色，以前常说人在loop中，那具体的角色应该是什么呢？审计者还是指挥官？作为指挥的层级是ceo还是leader？
![循环工程](/assets/images/posts/2026/feishu-CnLpbLuPLoSP.png)

其实从这些角度看演进，loop 工程也同样是有人已经做过类似的，而后才提出这么个名词概念，当然我们深究下去是不同的。在单轮对话中，比如claude code是能实现这种loop，而其中提到的计划 评估 反思 记忆 循环，这过程其实都是agent自己来决定要不要走，或者说利用skill启动这个工作流。没错这可以作为工作流给到agent开启这个循环。比如codex的goal模式，会以一个目标不断循环迭代下去。
> 这里可引申出claude code最近提出来的动态工作流，让agent自己沉淀和实践工作流，且大部分最好是能脚本工具化，还需再阅读。

其实这过程我在参加一些比赛中经常用到，不过是手动提示词启动的，大部分也还是需要我来审查和提示启动下一波循环，所以这就像是一个需要提示词进入下一步循环的智能体，不能完全自主进入循环。但我认为目前主要还是成本问题导致不能走这种自我驱动的闭环。单独一轮闭环通过提示当然可以开启，而多轮循环的效能不确定我认为是一个大的阻碍点。claude code创始人当然可以经常使用这个loop，因为他们不缺token。
在美团黑客松比赛之前，我还在研究深度学习的视觉，其实对于我们"低阶"的算法应用工程师而言，科研的操作目前已经差不多固化为调研文献-》看适配度-〉是否值得复现或融入-〉拿来试-》结果不行-〉分析问题-》调参/再调研。
这个能覆盖大部分情况，当我们用这套流程控制大多数变数后，才会再次去思考更根源的问题-数据，毕竟数据成本大部分情况很高，尤其自身业务的特殊数据。那么其实在这些过程中，起到关键作用的就是数据和模型，模型侧包含结构、训练参数、函数设计、架构范式，这些全都可以通过调研然后思考得到，AI可以辅助迭代。数据侧一般在一开始会考虑清楚，而后出现顽固问题的时候才会考虑。那么快速迭代在深度学习当中才是真理，你的大模型训练够快迭代够快就能更快发现问题解决问题就能得到更快速的scaleup。当然这也只是我的观点。
这其实前置的迭代过程就是一个自动求解器，我们需要调研规划实施复盘改进留痕沉淀再迭代。整个过程我都尝试做成了一个雏形。
有兴趣可以看https://github.com/Qnh233/SegLoop

上面这一套下来，总结一下就是具有明确目标，整个工作流可用AI替代或辅助，存在循环迭代求解的需求，都可以设计出一个loop系统，或者说工作流，我们的目标需要明确，需要有稳定的评测集和评测器，给Agent一个安稳安全可控的环境进行自主迭代。这里一个关键是目标如果是语义模糊的怎么办，没有固定评测怎么办？遇事不决AI取决。当然是和AI交互出语义目标，用LLM as judge来评估。

# 与k8s的联系
以我以前的视角来看，k8s就是一个自动运维工具，是一个容器编排平台。
其实要更复杂更优雅。
**Reconciliation Loop. Controller  ｜ Agent → Controller ｜ Workflow → Reconciliation Loop**
协调循环器就一个任务，一直看现在的状态是不是期望状态，如果不是就调整。这就可以类比到现在的目标是不是达到期望目标？这是一个goal，是一个循环是否继续的判断条件。Prompt/Desired State → Goal
k8s中也是观测，比对然后行动act再进入循环。Chat → State
这里核心循环是相似的。
loop工程中的行动就可以简化为一次次的chat，chat保存下来的迭代记忆memory 就是context。工作流即一轮循环要做什么就说前面的**Reconciliation Loop。**要不要继续怎么继续都是controller决定，即Agent。

总体而言，就是核心工程的一个智慧，guides前馈的约束控制，sensors的反馈纠正控制。
其实k8s有些已经开始了agent的应用实践。包括最近为参加的一个比赛也是智能运维的项目，不过没有涉及k8s。

## 关键区别
除了上述的节点区别工作流区别，其实最关键的区别还是前面我们一直讲的，确定性区别。
k8s的运维中你的期望状态是确定的，比如cpu占用率，延时等有阈值，不满足就去对应的act，且act也是固定的，我们知道如何做能做到什么效果。这就是在控制轮中的plant model 被控对象模型，我们能够结构化这个对象，有清晰具体确定的操作对应于结果，是可预测的。
搬到LLM里，除了本身的黑盒概率云，还有就是大部分任务目标只会是一个prompt，一个语义的目标，下面的act可能完全不同，甚至可能会背道而驰。
