### 核心问题逐一拆解

**关于 LangChain / LangGraph**

对于这个 MVP，**不建议引入 LangChain，但有条件引入 LangGraph**。

LangChain 的问题在于：它是"为复杂 Prompt 链而生"的框架，MVP 阶段的 Prompt 逻辑完全可以用原生 Python 字符串模板 + 直接 API 调用替代，引入 LangChain 只会增加调试成本和启动门槛，而不带来任何实质增益。

LangGraph 则是另一回事。主观题批改天然是一个**多步骤、有状态的工作流**：OCR → 解析题型 → 分发批改逻辑（客观/主观分叉） → 生成评语 → 输出结果。如果你觉得这个流程后期需要加步骤（比如先检索知识点、再生成个性化建议），LangGraph 的状态机模型非常合适。**但 MVP 阶段建议先用原生 Python if-else 分支走完流程，等逻辑稳定后再迁移到 LangGraph**，避免过早引入复杂度。

---

**主观题：语义匹配 vs 大模型评分**

文档一没有明确写主观题的技术实现，这是最需要补充的点。

**单纯语义匹配（embedding 相似度）不够用**，原因很具体：填空题"牛顿第一定律"和"惯性定律"语义相近，embedding 能判对；但作文题里，学生写了 500 字，你根本没有一个标准答案可以和它做相似度比较。更关键的是，数学解答题需要**判断过程步骤是否正确**，这是纯语义相似度完全无法做到的事。

## 优化后的关键变更汇总

**技术栈新增/替换：**

原方案的 Levenshtein 单一策略替换为三级分流。新增 sympy 处理数学等价（pip install sympy，零学习成本），用 sentence-transformers 或直接调用豆包的 embedding 接口做语义相似度兜底。这两个改动加起来不超过 100 行代码，但覆盖面从"能跑通"升级到"实际可演示不翻车"。

**主观题补全方案：**

文档一的最大缺口是主观题没有具体实现路径。核心补充是：给豆包设计两套固定 Prompt 模板（数学解答题一套、作文一套），**要求大模型只输出 JSON 结构**（得分 + 扣分原因 + 知识点标签），然后用 Python 解析 JSON 渲染到前端。这样批改结果可追溯、可展示，完全满足两个比赛的评分要求。

**LangGraph 的正确时机：**

题型解析和分流是整个系统最核心的"路由器"逻辑。MVP 阶段用 if-else 判断，够用且透明易调试。等主观题 Prompt 稳定、学情分析逻辑固化之后，把这一层迁移到 LangGraph 的 StateGraph，每个节点对应一个批改步骤，天然支持未来加入"人工复核节点"——这正好对应图一比赛里"支持人工复核与结果修正机制"的硬性要求，一步到位。

**两周执行节奏建议：**

按原文档的阶段划分节奏没问题，但优先级排序稍作调整：第一个能端到端跑通的 Demo 应该是"数学客观题（选择+填空）"，因为它同时验证了 OCR 管道和批改逻辑两条关键路径，失败点最容易定位；作文批改放在第二周前半段，重点时间花在 Prompt 调优上，而不是代码本身。

现在我已经充分了解 PocketFlow 的核心设计。让我给出完整分析：---

### PocketFlow 是否适合作为 MVP 架构？结论：强烈推荐

PocketFlow 的核心抽象就三个东西：**Node（节点）、Flow（图连接）、SharedStore（共享字典）**。你的批改流水线天然就是一个图：OCR → 题型解析 → 分叉批改 → 评语生成 → 输出。这和 PocketFlow 的设计哲学完全吻合。

相比原方案的 if-else Python 分支，换成 PocketFlow 的具体收益：

**零依赖、56KB**。对比 LangChain 的 166MB 和 LangGraph 的 51MB，在技术文档里写"自研轻量编排框架（基于开源 PocketFlow 100 行核心）"这个表述对评委来说非常加分，显得架构有审美。

**HITL 节点是现成的**。PocketFlow cookbook 里有完整的 pocketflow-cli-hitl 示例，低置信度批改结果自动进入人工复核队列，这正好对应图一比赛"支持人工复核与结果修正机制"的硬性要求，直接拿来用。

**BatchFlow + AsyncParallelBatchFlow**。一班 40 份作业不是逐一处理的，而是并发送出 OCR 和 LLM 请求，官方 cookbook 实测有约 8 倍提速。这个数字可以直接写进技术文档的性能测试部分。

**每个 Node 的三段式生命周期**（prep → exec → post）天然把"读数据、调API、写结果"三件事分离，出错时定位极快，对零基础来说调试体验远好于手写 if-else 管道。

---

### MVP 之后的优化路径

**第一优先级：提升批改质量**

引入 Majority Vote 模式，对主观题调用大模型三次取多数结果，PocketFlow 有现成 cookbook 示例。这在技术文档里可以体现为"评分一致性机制"，对评委非常有说服力。数学解答题还可以加一个 Supervisor Node：在主批改 Node 之后跑一个独立的验证 Node 检查评分逻辑是否自洽，不通过就重跑，这正是 PocketFlow 的 pocketflow-supervisor 示例。

**第二优先级：RAG 知识库**

把中小学学科知识点大纲向量化存入本地 vector store（用 SQLite + sqlite-vec 插件，零额外服务），批改时每道题先检索最相关的知识点再喂给大模型，评语里的知识点定位会精准很多。这是从"能用"升级到"好用"最有效的单一改动。PocketFlow 的 RAG 示例可以直接参考。

**第三优先级：学生记忆 Node**

加一个 StudentMemoryNode，把每次批改结果写入学生的长期错误记录，下次批改时把该学生过去三次的薄弱点作为上下文喂给评语生成节点，让评语从"这道题错了因为..."升级到"你在分数/分母相关题上已经连续出现类似错误，建议..."。这对应两个比赛都要求的"个人知识画像"，是差异化亮点。

**第四优先级：模型替换灵活性**

因为 PocketFlow 没有任何 vendor wrapper，你的 LLM 调用函数完全自己写。这意味着可以无缝切换豆包、通义、GPT-4o 甚至本地部署的 Qwen。后期如果要做横向对比测试（A/B 测试哪个模型评语更受教师认可），只需换一行 API 调用，整个 Flow 不动。

**长期方向：A2A 多 Agent 架构**

PocketFlow 现在支持 A2A 协议，意味着可以做"语文 Agent + 数学 Agent + 学情分析 Agent"三个独立进程互相通信，各自维护专属的知识库和 Prompt 策略。这不是 MVP 要做的事，但架构上从第一天就没有堵死这条路，这正是选 PocketFlow 而不是硬写 if-else 的长期价值所在。
